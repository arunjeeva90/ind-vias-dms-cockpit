# Unified state mapping and contour-line face mesh for DMS Cockpit

This change eliminates scattered state-mapping logic from five cockpit components, consolidating it in a single `driverStateMapping.ts` utility that maps `DriverState` + confidence to a five-level `OverallState`. The critical ADAS integration display bug (showing 7827% due to a double ×100 on `integrationScore`) is fixed. The RightHeadPanel is completely rewritten — the 200+ triangle polygon mesh is replaced with a layered SVG contour-line visualization (7 depth layers of path elements). Smaller refinements in CenterVideoPanel, LeftConsole, DecisionCard, and BottomTimeline tighten bounding boxes, add iris dots, improve explanation text, and standardize output contract labels.

**Watch for:** The `DEGRADED` threshold at confidence < 0.3 swallows all other state information (confirmed), meaning a brief camera occlusion could mask a genuine DROWSY/DANGER state until confidence recovers. The BottomTimeline stores `confidence` at state-transition time only, so timeline replay chips can show stale confidence values if confidence drifts without a state change (confirmed). The gaze cone geometry in RightHeadPanel points upward regardless of actual gaze direction in some quadrants (likely).

## High-level view

The shared utility introduces a two-input mapping: raw `DriverState` enum plus `confidence.overall` produces one of five `OverallState` values. The confidence gate at 0.3 overrides everything — if the system is unsure, it reports DEGRADED rather than a potentially false positive. All five cockpit panels now import from this single source, so label/color drift between panels is structurally impossible. Camera health derivation similarly centralizes, reading both `faceDetected` and `quality` rather than just the boolean.

The RightHeadPanel rewrite swaps rendering strategy entirely: from a `FACE_MESH_TRIANGLES` array of filled polygons to seven concentric layers of SVG `<path>` elements that trace facial contours. CSS 3D rotation with clamping at ±35° provides spatial feedback. A gaze cone polygon originates from the midpoint between both eye landmarks, colored by on-road status — but the cone geometry has a directional bias that makes it misleading for downward gaze.

The CenterVideoPanel refinements are cosmetic-structural: tighter face bounding box (70×85 down from 80×95), iris/pupil dots inside eye regions, a confidence pill label near the bbox corner, and a narrower neck path. The LeftConsole OUTPUT CONTRACT section now uses the exact underscore-separated uppercase labels expected by downstream consumers. DecisionCard gains better `whyExplanation` prose per state and confirms the integrationScore fix.

## State mapping consolidation and the confidence gate

The confidence gate at 0.3 sits above the state switch — any signal below that threshold returns DEGRADED before the driver state is even consulted. A driver in `DriverState.FATIGUED` with confidence 0.28 (perhaps due to momentary IR glare) will display as DEGRADED in every panel, including the DecisionCard's HMI action recommendation. The system effectively fails-silent on the actual state rather than failing-safe. In a production DMS, the convention is typically to hold the last known dangerous state during confidence drops rather than suppress it.

```typescript
// driverStateMapping.ts line 30
export function mapDriverState(state: DriverState, confidence: number): OverallState {
  if (confidence < 0.3) return 'DEGRADED';  // all state info lost here
  ...
}
```

## RightHeadPanel gaze cone direction bias

The gaze cone is constructed as a `<polygon>` with three vertices: the midpoint between both eyes, and two points offset by `gazeX ± 18` and `gazeY - 45`. The Y offset is always `-45`, so the cone always extends upward in SVG coordinate space:

```typescript
points={`${gazeMidX},${gazeMidY} ${gazeMidX + gazeX - 18},${gazeMidY + gazeY - 45} ${gazeMidX + gazeX + 18},${gazeMidY + gazeY - 45}`}
```

When `data.gaze.y > 0.5` (driver looking downward), `gazeY` becomes positive via `(data.gaze.y - 0.5) * 50`, but the polygon vertex is still at `gazeMidY + gazeY - 45` — which for moderate downward gaze (gazeY around 10-15) still places the tip above the eyes. Only extreme downward gaze (gazeY > 45, requiring `data.gaze.y > 1.4`, outside the 0-1 range) would flip the cone below the origin. For the common "looking at phone in lap" scenario, the cone points the wrong direction.

## BottomTimeline state history confidence capture

The `StateHistoryEntry` captures `confidence` only inside the condition `lastState.state !== data.driverState` — so confidence is sampled only when state changes. If a driver is in DROWSY for 30 seconds while confidence drops from 0.8 to 0.25, the timeline chip will still show MONITOR (mapped from DROWSY + 0.8) because the stored confidence is stale. The chip is historically accurate for when the transition happened, but won't reflect real-time confidence degradation for the most recent chip still on screen. A fix would be to update the last entry's confidence on each tick, or to re-derive chip colors from current confidence rather than stored values.

## DecisionCard degraded-reason threshold mismatch

`getDegradedReason` triggers when `confidence.overall < 0.5`, but `mapDriverState` only returns DEGRADED when confidence < 0.3. Between 0.3 and 0.5 the state badge might show WARNING (from DISTRACTED) while the degraded reason simultaneously shows "Low signal quality" or "Eyes not visible". If intentional as an early-warning signal, it should be documented; if not, align the thresholds.
