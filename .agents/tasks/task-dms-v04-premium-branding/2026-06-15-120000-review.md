# Premium branding and OEM engineering panel overhaul for DMS cockpit

This change transforms the IND-VIAS DualSight DMS Cockpit from a functional prototype into a branded OEM-quality engineering dashboard. It adds LTEPS company branding (brand colors, gradient accents, animated glow), a circuit-trace decorative background pattern, 29-point facial landmark visualization in the center video panel, a 50+ element dense wireframe head mesh in the right panel, a full OEM engineering left console with state ladder and output contract section, enriched decision card logic, and a state-history timeline. The approach is purely visual and informational: no business logic, telemetry pipeline, or API contracts changed.

**Watch for:** Duplicate circuit-trace background rendered twice (body::before AND Dashboard overlay div) — possible performance concern on low-end devices (confirmed). The `useEffect` in BottomTimeline lists `[data]` as its dependency — since `data` is a new object reference on every render from the telemetry hook, this effect fires every render regardless of whether values actually changed (confirmed). Output contract thresholds in LeftConsole can contradict the state badge derived from `DriverState` enum (confirmed).

## High-level view

The branding layer introduces five LTEPS color tokens into Tailwind and uses them at very low opacity in the status bar gradient accent, section-header line art, and a continuously-animated text glow on the "LTEPS" mark. The circuit-trace pattern is implemented twice: once as a CSS `body::before` pseudo-element (200px tile, 4% opacity) and once as an inline-style `div` in Dashboard.tsx (300px tile, 3.5% opacity), stacking to ~7% combined. On embedded automotive hardware this double-compositing is worth profiling.

The center video panel is now a full simulated IR camera frame with vignette, scanlines, noise texture, facial landmarks in five anatomical groups, eye ROI highlights, and RGB head-pose axes. All landmark positions are offset from a single `faceCx/faceCy` anchor. The `feTurbulence`-based noise filter is the most expensive element — it generates per-pixel procedural noise at 800x500 resolution on every paint.

The right head panel uses a dense SVG wireframe inside a CSS 3D perspective container that rotates with clamped yaw/pitch/roll. The gaze cone originates from the midpoint between both eyes. A new status info panel below the mesh shows hardcoded "Tracking Locked" and "Head Pose Valid" text that doesn't react to telemetry state.

The left console's output contract section derives four formal DMS states using local threshold constants (30/60/80) that are independent from the `DriverState` enum used by the state badge. These two derivation paths can produce contradictory displays when a driver is at score 35 (output says MONITOR, badge says NORMAL).

The bottom timeline introduces a state-history chip row tracked via a ref, but the underlying `useEffect` fires on every render due to referential inequality of `data`, meaning the history buffer accumulates points faster than telemetry actually updates.

## Duplicate circuit-trace background pattern

The circuit-trace decorative effect is rendered in two independent layers. In `src/index.css`, a `body::before` pseudo-element fills the viewport with a 200x200 tiling SVG at 4% opacity. In `src/components/Dashboard.tsx`, an absolutely-positioned div renders a different 300x300 tiling SVG at 3.5% opacity.

The two SVGs have different path geometries, so they layer producing a combined ~7% opacity pattern with visual moiré. The browser composites two full-viewport tiled SVG backgrounds on every repaint. For a dashboard designed to run on embedded automotive hardware, this doubles GPU texture memory for a purely decorative effect.

If both are intentional for depth, promote the `body::before` to its own compositing layer with `will-change: transform`. If one is a leftover from iteration, remove it.

## useEffect dependency and render-triggered accumulation

```tsx
// BottomTimeline.tsx
useEffect(() => {
  const point: TimelineDataPoint = { ... };
  historyRef.current = [...historyRef.current, point].slice(-MAX_POINTS);
  // ...
}, [data]);
```

The `data` object is a new reference on every render from `useTelemetry`. The effect runs unconditionally on every render — functionally equivalent to no dependency array. If the parent re-renders for reasons other than telemetry updates (window resize, React concurrent features), spurious data points get appended to the history buffer, inflating the timeline with duplicate readings.

Fix: compare `data.timestamp` to a `prevTimestampRef` and skip the append if unchanged.

## State-history key stability

```tsx
{stateHistory.map((entry) => (
  <span key={entry.time} ...>{getStateChipLabel(entry.state)}</span>
))}
```

`entry.time` is `counterRef.current` at append time. Since the counter increments inside the same `useEffect` that fires on every render (issue #2 above), rapid re-renders could produce entries with adjacent but theoretically collision-prone values. In React strict mode (dev), the effect double-fires within the same commit, and `counterRef.current` is shared mutable state — both invocations read and increment it, but the second may or may not see the first's increment depending on React's batching. A UUID or separate monotonic ID ref would eliminate the ambiguity.

## Hardcoded status values that diverge from telemetry

- **LeftConsole** line 163: `Calibration` always reads "OK" regardless of any telemetry signal.
- **RightHeadPanel** status info grid: "Head Pose Valid" and "Tracking Locked" are static green text. When `confidence.faceDetected` is false, the panel still shows these success indicators alongside a correctly-derived "Gaze: Off Road" warning — a contradictory signal mix.

The `Eye L: OPEN | Eye R: OPEN` display is derived from the single `confidence.eyesVisible` boolean, so both eyes always show the same state. The "L / R" framing implies independent per-eye tracking that the telemetry model doesn't support.

## SVG filter compositing load

CenterVideoPanel applies in a single pass:

- `feTurbulence` fractalNoise (per-pixel procedural generation across 800x500 = 400K pixels)
- Three separate `feGaussianBlur` + `feColorMatrix` chains for landmark glow groups
- `feGaussianBlur` on the face bounding box
- Full-viewport scanline pattern fill
- Full-viewport radial gradient vignette

On Qualcomm/NXP SoCs typical of DMS hardware, the `feTurbulence` filter alone can cost 5-10ms per frame. The noise rect has `opacity="0.3"` and the result after `feComponentTransfer` slope 0.04 is barely visible. Removing the noise filter or replacing with a static PNG data-URI would eliminate the most expensive compositing operation with negligible visual impact.

## Output contract threshold mismatch with state badge

The state badge in LeftConsole derives from `mapDriverState(data.driverState, data.confidence.overall)` — it maps the `DriverState` enum directly (ATTENTIVE -> NORMAL, DROWSY -> MONITOR, etc.).

The output contract section independently re-derives states from raw scores:

```tsx
function deriveDrowsinessState(score: number) {
  if (score >= 80) return 'DANGER';
  if (score >= 60) return 'WARNING';
  if (score >= 30) return 'MONITOR';
  return 'NORMAL';
}
```

Consider a scenario: `data.driverState` is `ATTENTIVE` (driver is not drowsy per the upstream DMS algorithm) but `data.drowsiness.score` is 35 (perhaps early-stage buildup that hasn't triggered a state transition yet). The badge shows NORMAL, but the output contract shows Drowsiness_State: MONITOR. The dashboard presents contradictory information about the same dimension. Either unify the derivation or label the output contract as "raw threshold" vs "algorithm output" to set expectations.

## DecisionCard static "why" explanations

The `whyExplanation` for DROWSY is always "Early drowsiness signs detected, monitoring escalation path" even when PERCLOS is at 80% and the system is in full fatigue territory that simply hasn't transitioned states yet. The word "early" is misleading when values are deep in the danger zone. Either make the text severity-aware (template in the actual metric values) or use neutral phrasing that doesn't imply a position on the severity scale.
