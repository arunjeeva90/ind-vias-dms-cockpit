# Live DMS telemetry receiver architecture

A new receiver subsystem sits between an external DMS pipeline (over WebSocket) and the existing cockpit dashboard. The change introduces a versioned wire-format contract (`DmsTelemetryMessage`), three receiver implementations (WebSocket with auto-reconnect, Dummy with a 6-state simulation machine, Replay for recorded sessions), runtime message validation, an adapter that maps the wire contract to the existing `DMSTelemetry` UI type, and a React hook that replaces the old `useTelemetry`. A Python PoC sender rounds out the integration story.

**Watch for:**
- The hook's live-to-dummy fallback path is dead code — `WebSocketDmsReceiver.connect()` never rejects, so the `.catch()` block never executes (confirmed).
- The existing `Dashboard.test.tsx` still mocks `useTelemetry`, which is no longer imported by Dashboard — the test will fail at runtime (confirmed).
- No unit tests were written for any of the new 1,700+ lines of application code (confirmed).

## High-level view

The receiver interface is minimal — connect/disconnect/subscribe/status — and each implementation (WebSocket, Dummy, Replay) fulfills it independently. The factory selects which to instantiate, but the hook's advertised fallback from LIVE_WS to DUMMY is broken: `connect()` resolves immediately without waiting for the WebSocket handshake, so the `.catch()` path that would create the dummy is unreachable.

The adapter (`toDashboardTelemetry`) handles 0–1→0–100 scaling consistently for `drowsiness.score`, `distraction.score`, `phoneSuspicion.confidence`, and `adasFusion.integrationScore`. The `confidence.overall` stays in 0–1 (matching UI's `* 100` display pattern). One mapping — `laneKeepAssist` aliased to `adasFusion.ready` — fabricates data rather than admitting absence, meaning the UI presents synthetic ADAS sub-states as if they were real telemetry.

The Dashboard.tsx change is a two-line import swap, but the existing test file still mocks the old hook path, breaking the test suite against this branch.

The validator checks structural presence and types for required fields but skips `scores.availability` and `scores.cameraQuality`, and performs no range checks. Out-of-range values (e.g., `"attention": 500`) flow through to the UI.

## Dead fallback in useDmsTelemetry

The hook's core selling point — "attempt LIVE_WS, fall back to DUMMY if WebSocket fails" — doesn't work as implemented. The `WebSocketDmsReceiver.connect()` method:

```typescript
async connect(): Promise<void> {
  this.shouldReconnect = true;
  this.error = null;
  this.attemptConnection();
}
```

This resolves immediately. The actual WebSocket connection attempt happens asynchronously inside `attemptConnection()`, and failures are handled via the internal `onclose`/`onerror` handlers which schedule reconnect. The promise never rejects.

In the hook:

```typescript
receiver.connect().catch(() => {
  if (mode === 'LIVE_WS') {
    const dummyReceiver = createDmsReceiver({ mode: 'DUMMY' });
    // ... fallback setup
  }
});
```

This `.catch()` block is unreachable. A user configuring `mode: 'LIVE_WS'` against a non-existent server will get perpetual reconnect attempts with no data, rather than graceful fallback to dummy.

Two viable fixes: (1) make `connect()` await the first handshake and reject on failure after a timeout, or (2) implement fallback via a status-polling check (e.g., "if no message received within 5 seconds of connect, spin up dummy").

Additionally, if the fallback code *were* reachable, it has a subscriber leak: the cleanup closure captures the `unsubscribe` function for the original LIVE_WS receiver, but the fallback dummy receiver's subscription (created inside the `.catch()`) is never unsubscribed on unmount.

## Broken Dashboard test

`Dashboard.test.tsx` contains:

```typescript
vi.mock('../../hooks/useTelemetry', () => ({
  useTelemetry: vi.fn(() => ({ ... })),
}));
```

But Dashboard now imports `useDmsTelemetry` from `../../hooks/useDmsTelemetry`. The Vitest mock intercepts the wrong module path. The unmocked hook will attempt to run, creating a real receiver and RAF loop inside a test environment.

Fix: update the mock to target `../../hooks/useDmsTelemetry` and mock `useDmsTelemetry` with the new return shape (which adds `receiverStatus` and `error`).

## Fabricated ADAS fields

The wire contract's `adasFusion` contains: `ready`, `integrationScore`, `forwardTtcSec`, `riskFusionState`. The UI's `ADASFusion` type expects: `ready`, `laneKeepAssist`, `collisionWarning`, `speedAdaptation`, `integrationScore`.

The adapter bridges this gap by fabricating values:

```typescript
const adasFusion: ADASFusion = {
  ready: message.adasFusion.ready,
  laneKeepAssist: message.adasFusion.ready,   // fabricated
  collisionWarning: true,                       // hardcoded
  speedAdaptation: message.adasFusion.ready,    // fabricated
  integrationScore: message.adasFusion.integrationScore * 100,
};
```

The dashboard presents "Lane Keep Assist: ON" and "Collision Warning: ON" without any real signal backing them. Either extend the wire contract to carry these fields, or have the UI distinguish "available" from "unknown/not reported."

## Validation gaps

The validator skips `scores.availability` and `scores.cameraQuality` — only `attention`, `drowsiness`, `distraction`, and `dmsConfidence` are type-checked. If a sender omits `availability` or `cameraQuality`, the message passes validation but produces `undefined` when the adapter reads `message.scores.cameraQuality`, propagating to the UI's quality-tier logic:

```typescript
quality: message.scores.cameraQuality > 0.7
  ? 'good'
  : message.scores.cameraQuality > 0.4
    ? 'fair'
    : 'poor',
```

With `undefined`, this evaluates to `'poor'` always — a fail-closed behavior (safe) but silently incorrect.

No range validation is performed. Scores outside 0–1 (e.g., a misbehaving sender emitting `"attention": 500`) pass validation and flow to the adapter where they become `50000%` in the UI.

## WebSocket handler cleanup race

When `disconnect()` is called while a WebSocket is in `CONNECTING` state, the `this.ws.close()` call triggers the `onclose` handler, which calls `scheduleReconnect()`. This is safe because `shouldReconnect` is set to `false` before `close()` is called, and `scheduleReconnect()` checks that flag. But the handlers remain attached to the closed socket instance, which is then set to `null`. If the browser fires a late `onerror` event on the closing socket, it will set `this.error` on an already-disconnected receiver — a minor state corruption unlikely to cause visible bugs but worth noting.
