# DMS telemetry dashboard — greenfield scaffold with dummy data generator

A complete React 18 + TypeScript + Vite SPA for visualizing Driver Monitoring System telemetry. The architecture separates data production (a `TelemetryProvider` interface with dummy and WebSocket implementations) from consumption (a `useTelemetry` hook that drives 9 dashboard panels). The dummy provider runs a state machine at 10 Hz with lerp-based smooth transitions between driver states (normal, drowsy, phone-check, distracted, recovering). A dark automotive theme is built with Tailwind custom colors. The project cannot be built in the current sandbox (npm install is blocked), so correctness relies on structural analysis and standalone tsc checks of pure .ts files.

**Watch for:** requestAnimationFrame loop in `useTelemetry` runs unconditionally even when no new data arrives, burning CPU for no visual change (confirmed). The seatbelt confidence field is typed 0-1 in the interface but the dummy provider emits 98-100, and the DMSConfidence `overall` field similarly drifts above 1.0 — consumers treat it as a percentage, not the 0-1 range the type documentation implies (confirmed). The WebSocket provider performs no schema validation on incoming messages — a malformed payload will silently propagate as a partial object through the entire UI (confirmed). GazeVisualization's trail accumulates via a ref mutated on every render, which can produce stale or skipped frames because the ref update races with React's reconciliation (likely).

## High-level view

The service layer defines a `TelemetryProvider` interface with a `DummyTelemetryProvider` (6-state machine, lerp smoothing, sine-wave noise) and a `WebSocketTelemetryProvider` (exponential backoff reconnection, zero message validation).

The `useTelemetry` hook stores incoming telemetry in a ref and copies it to state inside a requestAnimationFrame loop, decoupling the 10 Hz data rate from the render rate. The rAF loop spins unconditionally regardless of whether new data has arrived, and the hook sets `connected: true` before the WebSocket has actually opened.

Nine visualization components consume telemetry props from a `Dashboard` layout — SVG arc gauges, radial gauges, circular progress, conditional glow effects.

Tests cover the dummy provider thoroughly (24 tests) but nothing exists for the WebSocket provider, the hook lifecycle, or any SVG gauge math.

<summary>Issues (7)</summary>

1. **rAF loop spins unconditionally** — `useTelemetry` runs `requestAnimationFrame` in a tight loop even when no new data has arrived. Gate the `setData` call behind a dirty flag or timestamp comparison so React doesn't re-render on every animation frame.
2. **Confidence/seatbelt value range mismatch** — `DMSConfidence.overall` and `SeatbeltStatus.confidence` are documented and typed as 0-1 in `dms.ts`, but the dummy provider emits values like 95 and 98. Either the type comments are wrong or the provider is. Align them and add range normalization in consumers.
3. **No WebSocket message validation** — `WebSocketTelemetryProvider.onmessage` does `JSON.parse` then casts directly to `DMSTelemetry`. A malformed server message will produce a partial object that crashes downstream components expecting all fields. Add a runtime shape check or at minimum guard required nested objects.
4. **Gaze trail ref mutation during render** — `GazeVisualization` pushes to `trailRef.current` inside a `useEffect` triggered by prop changes. Because the ref is read in the same render pass that schedules the effect, the SVG renders trail data one frame behind the current gaze dot.
5. **Single-callback limitation in providers** — `onData` overwrites the previous callback. If multiple consumers subscribe (unlikely now, possible later), only the last one receives data. A listener-set or observable pattern would be safer.
6. **No WebSocket provider tests** — Reconnection logic (backoff timing, max-attempts cutoff, cleanup of timers on disconnect) is untested. These are the most likely places for resource leaks.
7. **`createTelemetryProvider` factory is unused** — The hook instantiates providers directly. The factory in `telemetry.ts` is dead code.

<summary>Details</summary>

## Value range mismatch between types and dummy provider

`seatbelt.confidence` is emitted as `98 + Math.random() * 2` (values 98-100) while the `SeatbeltStatus` interface documents it as 0-1 confidence. `DMSConfidence.overall` is computed as `92 + smoothNoise(t, 0.1) * 5` (values ~87-97) while the type says 0-1. Components like `DMSConfidence.tsx` consume it with `Math.min(Math.max(confidence.overall, 0), 100)` and render as a percentage, so the UI works today. But anyone connecting a real backend that emits the documented 0-1 values would get a meter stuck at 1%. The type contract and the producer must agree — either update the types to document percentage ranges or normalize in the provider.

## useTelemetry hook — rAF loop and premature connected state

The hook runs `requestAnimationFrame` calling `setData(latestDataRef.current)` on every frame (~60 fps). The dummy provider emits at 10 Hz, so ~50 of 60 frames call `setData` with the same object reference. React skips re-render for same references, but if any consumer ever spreads the telemetry object, every frame triggers a full re-render of 9 components. The rAF loop never pauses — `cancelAnimationFrame` only fires on unmount, so CPU cost persists in background tabs. A dirty flag set in the `onData` callback and cleared after `setData` would eliminate wasted frames.

The hook sets `connected: true` immediately after `provider.connect()`. For WebSocket mode, this means the UI shows "LIVE" before the socket opens, then flickers to "DISCONNECTED" if the connection fails, then back to "LIVE" on the next status poll (1s interval). The provider's `isConnected()` should be the single source of truth from the start.

## WebSocket provider — no payload validation, disconnect race

`onmessage` does `JSON.parse` then casts directly with `as DMSTelemetry`. If the server sends `{"type":"heartbeat"}` or a partial payload missing `headPose`, the callback receives an object that will throw when components access `data.headPose.yaw`. The try-catch catches parse errors but not schema violations. A guard checking for required top-level keys would prevent propagation of bad data.

If `disconnect()` is called while the WebSocket is in CONNECTING state, the socket's `onopen` will still fire and set `this.connected = true` after the provider thinks it's disconnected. Guarding `onopen` with `if (!this.shouldReconnect) { this.ws.close(); return; }` would close this race.

## Test coverage gaps

Not tested: WebSocket provider reconnection and backoff behavior — the most likely place for timer leaks. Not tested: `useTelemetry` hook lifecycle (connect on mount, disconnect on unmount, mode switching). Not tested: any visualization component beyond DriverStatePanel. The SVG gauge math in HeadPoseGauge and DistractionIndicator involves polar-to-cartesian conversions that are easy to get wrong but have no coverage.

## Unused factory function

`telemetry.ts` exports `createTelemetryProvider(mode, config)` but the hook imports providers directly. Either the hook should use the factory (cleaner DI, easier testing) or the factory should be removed to reduce dead code.
