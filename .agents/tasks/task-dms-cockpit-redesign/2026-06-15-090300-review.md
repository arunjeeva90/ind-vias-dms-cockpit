# DMS Cockpit UI Redesign — Automotive Console Layout

The dashboard was rewritten from a responsive card-grid into a fixed-size automotive cockpit console. Six new components replace the nine old card components, organized into a CSS Grid layout targeting 1920x1080 without scrolling. The design uses a `grid-rows-[auto_1fr_auto]` / `grid-cols-[280px_1fr_320px]` structure with a shared `DMSTelemetry` prop flowing from `useTelemetry` through Dashboard into each child.

Watch for: BottomTimeline runs an unguarded `useEffect` (no dependency array) that mutates refs on every render, creating a data-accumulation bug that fires extra points under React 18 StrictMode (confirmed). The StatusBar FPS counter is hardcoded to 30 and never reflects actual frame rate (confirmed). RightHeadPanel will render the wireframe as an unreadable thin line when head yaw exceeds ~50 degrees (likely, given the provider clamps at +-90).

## High-level view

The grid layout is sound for a fixed 1920x1080 viewport — column widths are pixel-locked for the side panels and `1fr` fills the center. The `h-screen` / `overflow-hidden` approach prevents body scroll but creates a risk: the BottomTimeline panel gets a fixed `h-[180px]` outside the auto-sized grid row, which works only because the middle row absorbs remaining space. On shorter viewports, the center panel clips without any visible indicator.

BottomTimeline is the most architecturally fragile component. It uses bare `useRef` buffers mutated inside a dependency-free `useEffect` to build a rolling time-series. Without a dependency array, the effect fires on every render — not just when telemetry changes. Any ancestor re-render pushes a spurious data point into the buffer.

The CenterVideoPanel and RightHeadPanel generate their gaze beam and axis indicators from raw telemetry degrees. The CSS `rotateY(${yaw}deg)` in RightHeadPanel will produce visually extreme rotation when yaw reaches values above ~45 degrees, rotating the flat SVG edge-on into illegibility. The dummy provider keeps values mild, but real telemetry could hit the +-90 degree clamp.

1. **BottomTimeline useEffect has no dependency array** — The effect runs on every render, not just when `data` changes. Under React 18 StrictMode it fires twice per data update, pushing duplicate points into the rolling buffer. Add `[data]` as the dependency array.

## BottomTimeline data accumulation under StrictMode

The `useEffect` in `BottomTimeline` has no dependency array:

```tsx
useEffect(() => {
  const point: TimelineDataPoint = { ... };
  historyRef.current = [...historyRef.current, point].slice(-MAX_POINTS);
  // ...
});
```

Without a dependency array, this effect runs after *every* render. The parent (`Dashboard`) re-renders whenever `useTelemetry` pushes new data via `setData`, so the sequence is: new telemetry -> Dashboard re-renders -> BottomTimeline re-renders -> effect fires -> pushes a data point. This is the intended behavior — one point per telemetry frame. But if any ancestor re-renders for reasons unrelated to telemetry (e.g., connection status polling), an extra spurious point gets pushed.

React 18 StrictMode compounds this: it double-invokes effects on mount with no cleanup to reverse the push, resulting in a duplicate entry at startup. The fix: `useEffect(() => { ... }, [data])`. This ensures exactly one buffer push per new telemetry frame regardless of unrelated re-renders.

## Hardcoded FPS display

```tsx
const fps = data ? 30 : 0;
```

The dummy provider fires at 10Hz. `useTelemetry` throttles via `requestAnimationFrame` (60Hz loop). The number 30 doesn't correspond to either rate. An operator interpreting this as a live metric would get misleading information. Either implement a frame-counter (track `data.timestamp` deltas over a sliding window) or change the label to indicate it's a target/nominal value.

## RightHeadPanel rotation extremes

```tsx
transform: `rotateY(${yaw}deg) rotateX(${-pitch}deg) rotateZ(${roll}deg)`
```

The dummy provider generates yaw up to ~40 degrees in `distracted` state and the `clamp` function allows +-90 on all axes. If real telemetry replaces the dummy, values near 90 degrees rotate the flat SVG edge-on, rendering it as a thin line. A visual clamp applied only to the CSS transform (e.g., `Math.max(-45, Math.min(45, yaw))`) would keep the wireframe legible regardless of input range without losing the raw values shown in the numeric readout below.

## Alert marker key instability

```tsx
{alerts.map((alert, i) => {
  const position = ((alert.time % MAX_POINTS) / MAX_POINTS) * 100;
  return (
    <div key={i} ... style={{ left: `${position}%` }} />
  );
})}
```

The alerts array is a sliding window (`.slice(-20)`). When a new alert is appended and the oldest is dropped, every remaining alert's index shifts by one. React re-uses DOM nodes by key, so shifted indices cause unnecessary re-paints. Using `key={alert.time}` provides stable identity since timestamps are monotonically increasing.

## Grid layout clip risk on sub-1080p viewports

On a 768px-tall screen, available height for the middle row is `768 - 48 (StatusBar) - 180 (timeline) = 540px`. The right panel stacks RightHeadPanel + DecisionCard vertically with `overflow-hidden`. The DecisionCard has no max-height and is not scrollable, so on shorter viewports the ADAS sub-indicators grid clips silently. Acceptable for the stated 1920x1080 target but worth noting as a known limitation.
